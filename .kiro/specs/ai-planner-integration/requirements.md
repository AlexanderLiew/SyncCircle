# Requirements Document

## Introduction

The AI Planner Integration feature adds intelligent scheduling capabilities to SyncCircle, enabling students to plan personal study sessions and coordinate group meetings with friends. The feature operates through a conversational chat interface backed by AI (OpenAI/Gemini), with all availability calculations and event creation handled server-side. The system supports two modes: Personal Planning (individual study sessions) and Group Planning (multi-participant meetings), both following a strict architecture where the backend retrieves authorized context, calculates valid availability, AI ranks options, and users explicitly accept before any event is created.

## Glossary

- **AI_Planner_Service**: The backend Lambda service that orchestrates planning requests, retrieves user context, calculates availability, invokes AI for ranking, and manages planning session lifecycle
- **Planning_Session**: A persistent backend record tracking the full lifecycle of a planning request from draft through confirmation or cancellation
- **Proposed_Time_Option**: A validated time slot generated by the AI_Planner_Service representing a feasible meeting or study time
- **Meeting_Invitation**: A record sent from a planning session creator to selected friends, requesting their acceptance of a scheduled group event
- **Calendar_Event**: A one-off scheduled event (distinct from recurring TimetableClass) persisted in the backend Events table
- **Availability_Calculator**: The backend module that computes free time periods by subtracting timetable classes and existing events from a user's schedule
- **Timetable_Privacy_Setting**: A per-user configuration controlling whether friends can view timetable data for group planning purposes
- **Free_Period**: A contiguous block of time during which a user has no timetable classes or calendar events scheduled
- **Creator**: The authenticated user who initiates a planning session
- **Participant**: A friend selected by the Creator to be included in a group planning session
- **Option_Card**: A frontend UI component displaying a Proposed_Time_Option with action buttons for accept, reject, or request alternatives

## Blocking Dependencies

The following capabilities do not currently exist in the backend and must be built as prerequisites or as part of this feature:

1. **Timetable Privacy Settings** — No backend mechanism exists for users to control whether friends can access their timetable for planning. The current `get-friend-timetable` handler only checks active friendship status.
2. **Backend Common Free-Time Calculation** — Free-slot computation currently runs on the frontend only. The AI Planner requires server-side calculation to ensure data integrity and security.
3. **Backend Event/Calendar-Event Creation API** — No backend persistence for one-off events exists. Only recurring weekly TimetableClass records are stored in the UserTimetables table.
4. **One-Off Event Data Model** — The system only stores recurring weekly classes (TimetableClass). A new Calendar_Event model with specific date/time (not day-of-week) is required.

## Requirements

### Requirement 1: Planning Session Creation — Personal Mode

**User Story:** As a student, I want to request a personal study session plan from the AI, so that I can find optimal times to study based on my schedule.

#### Acceptance Criteria

1. WHEN the Creator sends a personal planning request via `POST /ai-planner/personal` with activity, duration, and date range, THE AI_Planner_Service SHALL create a Planning_Session with mode "personal" and status "generating"
2. WHEN a personal Planning_Session is created, THE AI_Planner_Service SHALL retrieve the Creator's timetable classes, existing Calendar_Events, tasks, timezone, and AI preferences from the backend data stores
3. WHEN the Creator's context is retrieved, THE Availability_Calculator SHALL compute all Free_Periods within the specified date range by subtracting timetable classes and existing Calendar_Events from the available hours (08:00–23:00 local time)
4. WHEN Free_Periods are computed, THE AI_Planner_Service SHALL invoke the AI model to rank and explain valid time slots, then store results as Proposed_Time_Options on the Planning_Session
5. WHEN Proposed_Time_Options are generated, THE AI_Planner_Service SHALL update the Planning_Session status to "options-generated" and return the options to the Creator
6. IF the Creator's timetable or event data cannot be retrieved, THEN THE AI_Planner_Service SHALL return an error response with code "CONTEXT_UNAVAILABLE" and set the Planning_Session status to "draft"

### Requirement 2: Planning Session Creation — Group Mode

**User Story:** As a student, I want to schedule a group study session with friends, so that we can find times when everyone is available.

#### Acceptance Criteria

1. WHEN the Creator sends a group planning request via `POST /ai-planner/group` with activity, duration, date range, and selectedParticipantUserIds, THE AI_Planner_Service SHALL create a Planning_Session with mode "group" and status "generating"
2. WHEN a group Planning_Session is created, THE AI_Planner_Service SHALL verify via the Friendships table that each selectedParticipantUserId has an active friendship with the Creator
3. IF any selectedParticipantUserId does not have an active friendship with the Creator, THEN THE AI_Planner_Service SHALL reject the request with error code "NOT_FRIENDS" and list the invalid user IDs
4. WHEN all friendships are verified, THE AI_Planner_Service SHALL check the Timetable_Privacy_Setting for each Participant to confirm timetable sharing is permitted for the Creator
5. IF any Participant has disabled timetable sharing with the Creator, THEN THE AI_Planner_Service SHALL exclude that Participant's timetable from the calculation and indicate in the response which Participants' availability could not be determined
6. WHEN authorized timetables are retrieved, THE Availability_Calculator SHALL compute common Free_Periods across the Creator and all consenting Participants within the specified date range
7. WHEN common Free_Periods are computed, THE AI_Planner_Service SHALL invoke the AI model to rank options considering the number of available participants, time-of-day preferences, and duration fit
8. IF no common Free_Periods exist for the requested duration within the date range, THEN THE AI_Planner_Service SHALL return a response indicating no availability with suggestions to extend the date range or reduce the duration

### Requirement 3: Option Acceptance and Event Creation

**User Story:** As a student, I want to accept a recommended time slot, so that a study event is created on my calendar.

#### Acceptance Criteria

1. WHEN the Creator sends `POST /planning-sessions/{sessionId}/accept-option` with an optionId, THE AI_Planner_Service SHALL revalidate the selected time slot against current timetable and event data
2. IF the revalidation detects a conflict (the time slot is no longer free), THEN THE AI_Planner_Service SHALL reject the acceptance with error code "SLOT_CONFLICT" and suggest the Creator request new options
3. WHEN revalidation succeeds for a personal Planning_Session, THE AI_Planner_Service SHALL create a Calendar_Event for the Creator, update the Planning_Session status to "confirmed", and return the created event
4. WHEN revalidation succeeds for a group Planning_Session, THE AI_Planner_Service SHALL create a Calendar_Event for the Creator, update the Planning_Session status to "creator-accepted", and generate Meeting_Invitations for each Participant
5. THE AI_Planner_Service SHALL persist each Calendar_Event with: eventId, userId, title, startDateTime (ISO 8601), endDateTime (ISO 8601), duration, location, planningSessionId, and createdAt
6. THE AI_Planner_Service SHALL NOT create any Calendar_Event without explicit acceptance from the Creator via the accept-option endpoint

### Requirement 4: Option Rejection and Regeneration

**User Story:** As a student, I want to reject a suggested time and get alternative options, so that I can find a time that works better for me.

#### Acceptance Criteria

1. WHEN the Creator sends `POST /planning-sessions/{sessionId}/reject-option` with an optionId, THE AI_Planner_Service SHALL mark the Proposed_Time_Option as rejected and record it in the Planning_Session's excludedOptions list
2. WHEN the Creator sends `POST /planning-sessions/{sessionId}/next-option`, THE AI_Planner_Service SHALL generate new Proposed_Time_Options excluding all previously rejected time slots
3. IF no additional valid time slots can be found after exclusions, THEN THE AI_Planner_Service SHALL return a response indicating no further options are available and suggest modifying constraints
4. WHEN new options are generated, THE AI_Planner_Service SHALL update the Planning_Session with the new Proposed_Time_Options and maintain status "options-generated"

### Requirement 5: Meeting Invitation Lifecycle

**User Story:** As a student invited to a group study session, I want to accept or reject the invitation, so that I can manage my schedule.

#### Acceptance Criteria

1. WHEN a group Planning_Session reaches status "creator-accepted", THE AI_Planner_Service SHALL create one Meeting_Invitation per Participant with status "pending" and fields: invitationId, planningSessionId, eventId, senderUserId (Creator), receiverUserId (Participant), createdAt
2. WHEN a Participant sends `POST /meeting-invitations/{invitationId}/accept`, THE AI_Planner_Service SHALL update the Meeting_Invitation status to "accepted", create a Calendar_Event for the Participant, and update the respondedAt timestamp
3. WHEN a Participant sends `POST /meeting-invitations/{invitationId}/reject`, THE AI_Planner_Service SHALL update the Meeting_Invitation status to "rejected" and update the respondedAt timestamp
4. WHEN all Meeting_Invitations for a Planning_Session have been responded to, THE AI_Planner_Service SHALL update the Planning_Session status to "confirmed" if at least one Participant accepted, or "rejected" if all Participants rejected
5. IF a Meeting_Invitation is not responded to within 72 hours, THEN THE AI_Planner_Service SHALL update the Meeting_Invitation status to "expired"
6. THE AI_Planner_Service SHALL NOT send Meeting_Invitations until the Creator has explicitly accepted a Proposed_Time_Option

### Requirement 6: Planning Session Cancellation

**User Story:** As a student, I want to cancel a planning session I created, so that I can discard plans I no longer need.

#### Acceptance Criteria

1. WHEN the Creator sends `POST /planning-sessions/{sessionId}/cancel`, THE AI_Planner_Service SHALL update the Planning_Session status to "cancelled"
2. WHEN a Planning_Session with associated pending Meeting_Invitations is cancelled, THE AI_Planner_Service SHALL update all pending Meeting_Invitations to status "cancelled"
3. WHEN a Planning_Session with associated Calendar_Events is cancelled, THE AI_Planner_Service SHALL delete the Calendar_Events for the Creator and all Participants who accepted
4. IF a non-Creator user attempts to cancel a Planning_Session, THEN THE AI_Planner_Service SHALL reject the request with error code "FORBIDDEN"

### Requirement 7: Planning Session and Invitation Retrieval

**User Story:** As a student, I want to view my planning sessions and pending invitations, so that I can track my scheduled and pending plans.

#### Acceptance Criteria

1. WHEN an authenticated user sends `GET /planning-sessions`, THE AI_Planner_Service SHALL return all Planning_Sessions where the user is the Creator, ordered by createdAt descending
2. WHEN an authenticated user sends `GET /planning-sessions/{sessionId}`, THE AI_Planner_Service SHALL return the full Planning_Session including generated options and status, only if the user is the Creator
3. WHEN an authenticated user sends `GET /meeting-invitations`, THE AI_Planner_Service SHALL return all Meeting_Invitations where the user is the receiverUserId, ordered by createdAt descending
4. WHEN an authenticated user sends `GET /meeting-invitations/{invitationId}`, THE AI_Planner_Service SHALL return the Meeting_Invitation details including the associated event information, only if the user is the receiverUserId
5. IF a user requests a Planning_Session or Meeting_Invitation they are not authorized to view, THEN THE AI_Planner_Service SHALL return error code "FORBIDDEN"

### Requirement 8: Timetable Privacy Settings

**User Story:** As a student, I want to control who can see my timetable for planning purposes, so that I maintain privacy over my schedule.

#### Acceptance Criteria

1. THE AI_Planner_Service SHALL support a Timetable_Privacy_Setting per user with values: "friends" (all active friends can view), "none" (no one can view for planning)
2. WHEN a user has not configured a Timetable_Privacy_Setting, THE AI_Planner_Service SHALL default to "friends" (opt-out model)
3. WHEN a group planning request includes a Participant whose Timetable_Privacy_Setting is "none", THE AI_Planner_Service SHALL exclude that Participant's schedule from availability calculations and indicate reduced accuracy in the response
4. THE AI_Planner_Service SHALL NOT expose individual timetable class details of Participants in group planning responses; only aggregated Free_Periods shared among all consenting Participants SHALL be returned

### Requirement 9: Calendar Event Data Model

**User Story:** As a student, I want my planned events stored persistently in the backend, so that they are considered in future planning and synced across devices.

#### Acceptance Criteria

1. THE AI_Planner_Service SHALL persist Calendar_Events in a DynamoDB table with partition key "userId" and sort key "startDateTime"
2. THE Calendar_Event record SHALL contain: eventId, userId, title, startDateTime (ISO 8601), endDateTime (ISO 8601), duration (minutes), location, planningSessionId, participantUserIds, status (active/cancelled), createdAt, updatedAt
3. WHEN the Availability_Calculator computes Free_Periods for a user, THE Availability_Calculator SHALL subtract both TimetableClass recurring slots and Calendar_Event one-off events from the available time
4. WHEN a Calendar_Event is created, THE AI_Planner_Service SHALL return the event data including eventId and timestamps to the frontend for display and optional Google Calendar sync

### Requirement 10: Backend Free-Time Calculation Service

**User Story:** As a student, I want availability calculations done on the server, so that my schedule data remains secure and calculations are authoritative.

#### Acceptance Criteria

1. THE Availability_Calculator SHALL compute Free_Periods on the backend by combining recurring TimetableClass entries (mapped to specific dates within the requested range) and one-off Calendar_Events
2. WHEN computing availability for a specific date, THE Availability_Calculator SHALL map each TimetableClass to that date if the class's dayOfWeek matches the date's weekday (Monday=0 through Friday=4)
3. THE Availability_Calculator SHALL only consider time between 08:00 and 23:00 local time as available, treating all other hours as unavailable
4. FOR ALL valid inputs, computing Free_Periods then scheduling an event within a returned Free_Period then recomputing Free_Periods SHALL show the new event's time slot as occupied (round-trip consistency)
5. THE Availability_Calculator SHALL merge overlapping or adjacent busy periods before computing Free_Periods, producing non-overlapping sorted free slots
6. WHEN computing group availability, THE Availability_Calculator SHALL intersect individual Free_Periods across all consenting Participants to produce common free slots

### Requirement 11: AI Integration and Ranking

**User Story:** As a student, I want the AI to explain why it recommends certain time slots, so that I can make informed scheduling decisions.

#### Acceptance Criteria

1. WHEN the AI_Planner_Service invokes the AI model, THE AI_Planner_Service SHALL send only validated Free_Periods (not raw timetable data) along with the user's activity description, duration requirement, and AI preferences (responseStyle, planningAggressiveness)
2. THE AI_Planner_Service SHALL receive from the AI model a ranked list of recommended time slots with natural-language explanations for each recommendation
3. WHEN the AI model fails to respond within 15 seconds, THE AI_Planner_Service SHALL return Free_Periods without AI ranking as fallback options and indicate that AI explanations are unavailable
4. THE AI_Planner_Service SHALL validate that every AI-recommended time slot falls within a previously computed Free_Period before presenting it to the user
5. IF the AI model returns a time slot that does not fall within a valid Free_Period, THEN THE AI_Planner_Service SHALL discard that recommendation and log a warning

### Requirement 12: Authentication and Authorization

**User Story:** As a student, I want all planning operations secured, so that no unauthorized user can access my schedule or create events on my behalf.

#### Acceptance Criteria

1. THE AI_Planner_Service SHALL require a valid Cognito JWT token in the Authorization header for all API endpoints
2. WHEN a request lacks a valid JWT token, THE AI_Planner_Service SHALL return HTTP 401 with error code "UNAUTHORIZED"
3. THE AI_Planner_Service SHALL extract the authenticated userId from the JWT `sub` claim and use it as the sole identifier for authorization decisions
4. THE AI_Planner_Service SHALL verify that the authenticated user is the Creator before allowing modifications to a Planning_Session (accept, reject, cancel, next-option)
5. THE AI_Planner_Service SHALL verify that the authenticated user is the receiverUserId before allowing responses to a Meeting_Invitation

### Requirement 13: Input Validation and Rate Limiting

**User Story:** As a system administrator, I want input validation and rate limiting on AI planning endpoints, so that the system remains stable and secure.

#### Acceptance Criteria

1. WHEN a planning request is received, THE AI_Planner_Service SHALL validate that duration is between 15 and 480 minutes
2. WHEN a planning request is received, THE AI_Planner_Service SHALL validate that dateRangeStart is not in the past and dateRangeEnd is within 30 days of dateRangeStart
3. WHEN a group planning request is received, THE AI_Planner_Service SHALL validate that selectedParticipantUserIds contains between 1 and 10 user IDs
4. IF any input validation fails, THEN THE AI_Planner_Service SHALL return HTTP 400 with error code "VALIDATION_ERROR" and specify the failing field
5. THE AI_Planner_Service SHALL enforce a rate limit of 5 planning requests per user per minute on the `POST /ai-planner/personal` and `POST /ai-planner/group` endpoints
6. IF the rate limit is exceeded, THEN THE AI_Planner_Service SHALL return HTTP 429 with error code "RATE_LIMITED"

### Requirement 14: Frontend Chat Interface Enhancement

**User Story:** As a student, I want a conversational interface that shows AI recommendations as interactive cards, so that I can easily review and act on scheduling suggestions.

#### Acceptance Criteria

1. WHEN the AI_Planner_Service returns Proposed_Time_Options, THE Frontend SHALL render each option as an Option_Card displaying: start time, end time, duration, location (if provided), participant names (for group mode), and AI explanation text
2. THE Frontend SHALL display an "Accept" button and a "Find Another Time" button on each Option_Card
3. WHEN the user selects "Accept" on an Option_Card, THE Frontend SHALL call `POST /planning-sessions/{sessionId}/accept-option` and display a confirmation state on success
4. WHEN the user selects "Find Another Time" on an Option_Card, THE Frontend SHALL call `POST /planning-sessions/{sessionId}/next-option` and render the new options
5. THE Frontend SHALL display a mode selector allowing the user to switch between "Personal" and "Plan with Friends" modes before initiating a planning request
6. WHILE in "Plan with Friends" mode, THE Frontend SHALL display the user's active friends list (from `GET /friends`) with checkboxes for participant selection

### Requirement 15: Frontend Meeting Invitations Section

**User Story:** As a student, I want to see and respond to meeting invitations within the AI Planner, so that I can manage group study requests in one place.

#### Acceptance Criteria

1. WHEN the user navigates to the AI Planner page, THE Frontend SHALL fetch pending Meeting_Invitations via `GET /meeting-invitations` and display a notification badge with the count of pending invitations
2. THE Frontend SHALL display each pending Meeting_Invitation with: event title, proposed time, duration, sender name, and participant list
3. WHEN the user taps "Accept" on a Meeting_Invitation, THE Frontend SHALL call `POST /meeting-invitations/{invitationId}/accept` and update the invitation status in the UI
4. WHEN the user taps "Reject" on a Meeting_Invitation, THE Frontend SHALL call `POST /meeting-invitations/{invitationId}/reject` and remove the invitation from the pending list

### Requirement 16: Frontend Empty and Error States

**User Story:** As a student, I want clear feedback when the AI Planner cannot function, so that I understand what action to take.

#### Acceptance Criteria

1. WHEN the user has no active friends and selects "Plan with Friends" mode, THE Frontend SHALL display an empty state message explaining that friends are required and provide a link to the Friends page
2. WHEN the Availability_Calculator returns no Free_Periods, THE Frontend SHALL display a message indicating no available time slots and suggest adjusting the date range or duration
3. WHEN the AI_Planner_Service returns an error, THE Frontend SHALL display the error message with a "Retry" button
4. WHEN the frontend cannot reach the AI_Planner_Service backend, THE Frontend SHALL display a connectivity error with guidance to check network connection

### Requirement 17: Google Calendar Sync for Created Events

**User Story:** As a student, I want events created by the AI Planner to sync to my Google Calendar, so that all my commitments appear in one place.

#### Acceptance Criteria

1. WHEN a Calendar_Event is successfully created, THE Frontend SHALL offer the user an option to sync the event to Google Calendar via the existing OAuth2 integration
2. WHEN the user confirms Google Calendar sync, THE Frontend SHALL create a Google Calendar event using the Calendar_Event's title, startDateTime, endDateTime, and location
3. IF the Google Calendar OAuth2 token is expired or unavailable, THEN THE Frontend SHALL prompt the user to re-authenticate with Google before syncing
4. THE Frontend SHALL NOT block event creation on Google Calendar sync; the Calendar_Event SHALL be persisted in the backend regardless of Google Calendar sync status

### Requirement 18: Data Security and Privacy in Group Planning

**User Story:** As a student, I want my private schedule details hidden from other participants, so that only aggregated availability is shared in group planning.

#### Acceptance Criteria

1. THE AI_Planner_Service SHALL NOT include individual TimetableClass titles, module codes, or locations of Participants in any group planning API response
2. THE AI_Planner_Service SHALL only return common Free_Period time ranges (start and end times) in group planning responses, without attribution to specific Participants
3. WHEN a Participant's Timetable_Privacy_Setting is "none", THE AI_Planner_Service SHALL indicate that the Participant's availability is "unknown" without revealing whether they have a timetable configured
4. THE AI_Planner_Service SHALL NOT pass Participant timetable details to the AI model; only computed Free_Periods SHALL be sent for ranking
